# Verification Stack — Active Work Block

## Objective
Execute the official MVP verification stack against the current top 3 candidates and measure whether the workflow can materially improve confidence.

This work block is the decisive test for Narrow B.

---

## Official stack to use
1. **Development.i**
2. **Planning scheme / local plan mapping**
3. **Overlay mapping**
4. **Application history**

Do not rely on:
- listing data for planning truth
- generic Google search as a primary source
- one-source verification
- `no evidence found` being treated as `no constraint`

---

## Inputs required
For each site:
- existing manual screen
- planning verification checklist
- verified-field operator worksheet
- timed verification test protocol
- verification source register
- planning source-gathering workflow

---

## Required outputs per site
Each pass must produce:
1. timed verification record
2. updated verified-field worksheet
3. recommendation before vs after
4. confidence before vs after
5. critical unknown count before vs after
6. note on what remained unresolved

---

## Success criteria
This work block is successful if one or more of the following occur:
- at least 1 site moves from **Low** to **Medium** confidence
- critical unknown count falls materially on a top-tier site
- verification time becomes commercially tolerable and repeatable
- the workflow proves it can extract decisive planning fields more reliably than before

---

## Failure criteria
This work block is a failure if:
- all 3 sites remain at low confidence with negligible field verification improvement
- verification time remains too high for a practical service model
- authoritative source access remains too brittle to support repeatable verification

---

## Decision gate after completion
### Continue Narrow B if:
- confidence uplift is real on at least one top-tier site
- decisive fields can be verified with repeatable effort
- total verification time is commercially acceptable

### Reposition as triage-first if:
- confidence does not improve materially
- unresolved critical fields remain dominant
- source access reliability remains the controlling bottleneck

---

## Operator discipline for this block
- use the official source order every time
- use fresh browser tabs when needed
- record actual time, not estimates
- mark unresolved fields explicitly as `unknown`
- do not overstate confidence

---

## Intended outcome
At the end of this work block, the project should be able to answer clearly:

> Is Narrow B operationally viable, or should the MVP stay positioned as a disciplined triage-and-escalation tool?
